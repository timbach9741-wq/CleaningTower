# Orchestration & Workflow Rules (Supreme Authority)

## 1. Immutable Code Policy (완료된 작업 보호)
- 이미 완료되어 Git에 커밋되었거나 `TODO.md`에서 완료(`[x]`) 처리된 기능들은 **절대 임의로 수정해서는 안 됩니다**.
- 새로운 기능을 추가하거나 기존 기능을 보완해야 할 경우, 기존 동작을 훼손하지 않는 선에서 `확장(Extension)`하는 방식을 취해야 합니다.
- 기존 핵심 로직(회원가입, 로그인, 관리자 리스트 등)을 불가피하게 수정해야 한다면, 작업 전에 반드시 사용자에게 "기존 코드 수정이 필요함"을 알리고 명시적인 승인을 받아야 합니다.

## 2. Plan Mode & Financial Integrity
- **Plan Mode**: 새로운 태스크를 시작하기 전, 현재 코드를 분석하고 어떻게 접근할지 단계별 계획을 먼저 제시합니다.
- 코드를 작성하기 전에 백업(Git Commit)이 되어 있는지 확인하여, 롤백이 가능하도록 환경을 안전하게 유지합니다.
- 비용이나 정산과 관련된 코드(Financial Integrity)는 절대 버그가 발생하지 않도록 교차 검증을 필수로 진행합니다.

## 3. Git Version Control
- 모든 유의미한 작업 단위(Task)가 완료되면 즉시 Git 커밋을 통해 기록을 영구 보존합니다.
- 커밋 메시지는 작업의 목적을 명확히 알 수 있게 작성하여, 추후 특정 시점으로 복구해야 할 때 헤매지 않도록 합니다.

## 4. Execution Process
1. 사용자 요청 확인
2. `TODO.md` 및 기존 히스토리 파악
3. (필요 시) `orchestration.md` 규칙에 어긋나지 않는지 점검
4. 변경 사항 설계 및 사전 공유
5. 코드 작업 및 테스트
6. 배포(Deploy) 및 Git 반영 (완료된 작업 봉인)
